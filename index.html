Mise en place d'un dictionnaire de donnée et d'un Modèle Conceptuel de Données pour une bibliothèque.

Dictionnaire de données et MCD pour une bibliothèque


Compétences mises en oeuvre pour ce projet :

  • Collecter, suivre les demandes
  • Traiter des demandes concernant les applications
  • Analyser les objectifs et modalités d'organisation d'un projet
  • planifier les activités

Le but de ce TP à été de se mettre en condition "réelle", en revenant à la base d'un projet. On imagine ici qu'une Bibliothèque nous a embauché pour developper un gestionnaire de base de données.

Nous nous sommes donc concentrés ici sur les deux premières étapes de tout projet incluant une base de donnée, et même si ces étapes ne comportent pas vraiment de code à proprement parler, elles restent néanmoins primordiales.

Le dictionnaire de données

La première étape est la réalisation d'un dictionnaire de donnée, On prend ici le temps de lister tout ce dont notre base de donnée va avoir besoin (type de données, relations entre ces dernieres...). Dans l'idéal, cette partie s'effectue en parallèle d'un dialogue auprès des clients qui sont aussi les futurs utilisateurs, et qui savent mieux que quiconque leurs besoins.

Ici une image de notre dictionnaire de données, on peut voir que les données sont déja pré-organisées et prêtes à passer à l'étape suivante :

Cette étape prend du temps, et ne doit pas être négligée car un oubli découvert plus tard dans la réalisation d'un projet (une fois que la base de donnée est créée et fonctionnelle par exemple) peut vite être un casse-tête à regler...

Le Modèle Conceptuel de Données (MCD)

Après avoir mis à plat l'ensemble des données dont nous allons avoir besoin, il est temps d'organiser ces dernieres, avec une modélisation conceptuelle de données.

Pour ce faire, nous avons utilisé l'application Looping, qui permet de créer des schémas de base de donnée, sur un modèle Entité/Relations (Tables et Relations entre les tables).

L'interêt de tels schémas et de pouvoir avoir un regard visuel sur notre base de donnée, et ainsi pouvoir plus facilement la structurer, en comparaison avec une création en script directement.


On regroupe ici nos differentes données dans des tables, qui vont être reliées entre elles via des relations.

L'application Looping permet ensuite de générer un script SQL depuis notre modèle, mais ce script doit généralement être édité pour etre parfaitement fonctionnel, ou pour convenir à notre façon de faire.

Conclusion

A la fin de ce TP, nous avons donc une base de donnée fonctionnelle, bien structurée et surtout adaptée aux besoins des utilisateurs, grace aux differentes étapes effectués. Ces dernières peuvent encore une fois sembler futiles, mais elles peuvent éviter de devoir faire marche-arrière alors que le projet est à un stade beaucoup plus avancé !

TP : Connexion BDD et login en C#

premières interactions avec une base de donnée



Connexion à une base de données "utilisateurs" en C# et cryptage des mots de passes.


×

Connexion BDD et login avec C#


Compétences mises en oeuvre pour ce projet :

  • Exploiter des normes et des standards
  • Vérifier les niveaux d'habilitation
  • Respecter les règles en terme de sécurité et confidentialité

Le but de ce TP à été d'apprendre à utiliser des outils (ici la librairie Mysql.data) de connexion à une base de donnée, et à pouvoir commencer à interagir avec de manière pratique et utile.

De plus, nous avons également vu, avec l'utilisation d'une base de donnée utilisateurs, les possibilités de cryptages des données ainsi que de bonnes pratiques de sécurité.

Création de la base de donnée

Nous avons pour commencer créé une simple base de donnée "login", comme suit :

La base de donnée ne comporte qu'une table "utilisateurs", avec, pour effectuer des tests, un premier élément "admin" pré-enregistré dedans.

Utilisation de MySql.Data

Après avoir créé un projet C#, nous avons importé la bibliothèque "MySql.Data", qui comprends un ensemble de classes et de methodes qui facilitent la connexion et l'interaction avec une base de donnée.

Connexion à la base de donnée

Ci-contre Nous avons créé une methode connect() qui prend en argument les differents parametres de connection à notre base de donnée.
Cette methode va en réalité encapsuler le constructeur de la classe MySqlConnection (qui sert a se connecter à une BDD) et factiliter son utilisation.
Notre méthode retourne un nouvelle instance de connexion, qui pourra être stocké dans une variable par exemple.


Créations de methodes

Il nous a fallu ensuite créer les différentes methodes permettant de créer un utilisateur, de connecter un utilisateur, et d'afficher les differents utilisateurs, chaque méthode utilisera l'objet de connexion établie plus haut.

createUser( )
Cette methode va permettre l'ajout d'un élément dans notre base de donnée avec les entrées utilisateur.
Le pseudo et le mot de passe sont récupérés dans des variables, et la variable password est crypté grace à Bcrypt, une autre bibliothèque que nous utilisons ici.
Ensuite, la requete est stocké dans une variable query, on remarque que les elements variables de celle-ci sont précédés d'un @, ce dernier est là pour prévenir les injections SQL : c'est seulement apres le stockage de la requete dans un nouvel objet _command que ces éléments variables prennent leurs vrai valeur.
Il ne reste plus ensuite qu'à executer la requête.

login( )
Cette methode permet à un utilisateur de se connecter grâce à ses identifiants.
Le pseudo et le mot de passe sont de nouveau récupérés dans des variables. Une requete est ensuite construite pour récuperer le mot de passe dans la base de donnée à l'identifiant indiqué. On note ici l'utilisation d'un objet _reader, qui permet la lecture de donnée, à l'inverse de la méthode utilisée plus haut.
Le code vérifie ensuite que la requete est bien valide (que l'utilisateur à bien été trouvé), puis compare, grâce à Bcrypt, le mot de passe entré avec le mot de passe crypté de la base de donnée. si ces conditions échouent, l'utilisateur en est averti.
On "ferme" ensuite l'objet _reader.

displayUser( )
Cette methode permet d'afficher les différents utilisateurs, elle est utile ici pour verifier que notre méthode createUser() est fonctionnelle.
Rien de bien compliqué ici : on effectue encore une fois une requete, que l'on va stocker dans un objet _reader, puis on va tour à tour afficher le contenu de cet objet (qui équivaut à chaque fois à une ligne de la table).

Conclusion

Nous avons au final créé une connexion à notre base de donnée, et créé plusieurs methodes permettant la création, et la connexion d'un utilisateur. De plus, nous avons permis à notre code d'être protégé des injections SQL, et avons renforcé la confidentialité des mots de passes en les cryptant.

TP : Création de blog en PHP (MVC)

Respect des standards de la profession



Création d'un blog en PHP en utilisant l'architecture Modèle/Vue/Controleur.


×

Blog PHP avec architecture MVC


Compétences mises en oeuvre pour ce projet :

  • Exploiter des normes et des standards
  • Assurer la continuité d'un service

Le but de ce TP à été de comprendre l'architecture MVC (Modèle/Vue/Controleur), qui est utilisé dans le milieu professionel (Laravel par exemple) pour sa modularité et sa facilité de maintenance.

Nous avons donc ici réalisé un blog, très simple, en reprenant des methodes de connexion à une base de donnée similaires à celles effectués dans le projet "Connexion à une base de donnée en C#" (voir liste des projets).
Peu à peu, nous avons fait évoluer notre code en le séparant progressivement en plusieurs fichiers, afin d'éviter la redondance et se diriger vers une architecture MVC.

L'architecture de base

Nous avons à chaque étape sauvegardé une version de notre projet, pour observer les changements, malheureusement je n'ai pas conservé la configuration initiale, mais il faut simplement retenir que l'ensemble du code (affichage et logique) etait contenu dans un seul fichier index.php, à l'inverse de ce que nous allons voir par la suite...

séparation intermédiaire

Nous avons séparé notre code contenu alors intégralement dans index.php en plusieurs blocs, respectivement dans les fichiers modele.php et vueAccueil.php

index.php
Ce fichier ne contient finalement plus que des requetes vers d'autres fichier, vers le modele.php pour les methodes propres a l'obtention des articles, et vers vueAccueil.php pour l'affichage de ces derniers. Il fait office ici de controleur, en assurant la liaison entre la vue et le modele.

modele.php
Ce fichier contiendra l'ensemble des méthodes de requete vers la base de donnée, méthodes qui retourneront un ensemble de données utilisable par le controleur, et donc par les vues.

vueAccueil.php
C'est le fichier qui contient (à ce stade) toute la partie HTML du projet. Il est appelé par index.php pour l'affichage des différentes données de la table.

séparation finale

Ici nous allons plus loin dans la séparation, avec l'apparition des fichiers controleur.php, params.php, gabarit.php ainsi que des vues supplémentaires. A ce stade, plusieurs méthodes ont été rajoutées dans notre modele.php, mais nous allons nous concentrer sur les autres nouveautés.

params.php
Ce fichier va contenir les paramètres relatif à la connexion vers notre base de donnée, plutot que d'avoir à intervenir sur la chaine de caractère qui gère la requete (voir fichier modele.php plus haut) on les stock ici dans des variables utilisables par les autres fichiers.

controleur.php
Le fichier remplace ici une grosse partie du travail d'index.php précédemment, c'est maintenant ici que les methodes de modele.php sont appelées et utilisées, tout comme les vues qui vont êtres utiles pour l'affichage.

Vues / vueAccueil.php
On remarque que ce fichier a été allégé : on à enlevé toute la partie HTML d'"enrobage" (doctype, head, header et footer). On ne garde ici que la partie propre à ce que l'utilisateur va voir sur l'accueil de notre blog. La même chose est appliquées aux autres vues ajoutées dans cette version finales, avec chacune un contenu distinct.
La particularité ici est que les données HTML vont êtres gérés par les méthodes ob_start() et ob_get_clean(), ces dernières vont détourner ces données pour les stocker dans une variable $contenu.

gabarit.php
Ce fichier est donc le "template" regroupant le code HTML commun à toutes les pages (doctype, head, header, footer). Le fichier contient dans son main la variable $contenu qui va recueillir le contenu différentes vues, qui ne contiendront alors que le code HTML relatif à leur page.

index.php
Le point d'entrée de notre blog fait maintenant office de "controleur principal". Il va ici surtout appliquer des conditions aux differents affichages possible, dans le but de gérer les erreurs potentielles (connexion impossible à la base de donnée, aucun articles disponibles, etc) et de les afficher en conservant l'interface utilisateur du site, via vueErreur.php.

Et voilà !
Après la création de sous-dossiers pour mieux organiser nos différents fichiers, on retrouve une architecture MVC propre et fonctionnelle.

Résultat

Conclusion

Au cours de ce TP, nous avons donc pu faire evoluer notre code vers une forme plus structuré, et nottamment mieux adaptée au travail en équipe, de part sa modularité.

Mon sujet.

J'ai décidé de choisir Tailwind CSS comme sujet de veille technologique : Je vous explique donc pourquoi mon choix s'est porté là dessus, et sa pertinence dans le developpement web.

Tailwind CSS est Un framework du langage CSS, qui a pour but de simplifier et surtout de rendre plus fluide l'intégration de styles CSS au sein d'un projet, en ayant la possibilité d'appliquer son style directement dans nos fichiers HTML et ce de manière "optimisée".

Un exemple de code CSS classique

Et le même code avec Tailwind CSS

Ce framework vient se proposer en alternative à des frameworks comme Bootstrap par exemple. Il est devenu depuis sa sortie en 2017 un outil très populaire auprès des développeurs, et j'estime que son apprentissage est une vrai plus-value d'un point de vue professionnel, d'où ma volonté de me renseigner sur ce sujet.

L'intêret

L'ajout de CSS dans un projet est souvent une tâche laborieuse, la pratique "standard" est de définir des classes à nos balises HTML, et ensuite, à l'interieur d'un fichier CSS, de joindre des directives de style à ces classe. Cette façon de faire est certe efficace pour les projets simples, mais dès que ce dernier se s'enrichi et que les besoins de design s'etoffent ou deviennent plus "situationnels", le ou les fichiers CSS deviennent vite chaotiques, et long.

Tailwind CSS va nous donner accès à une librairie de classes "utilitaires" CSS qui correspondent toutes à une directive css en particulier, L'ensemble de ces classes sont accessibles depuis leur site officiel (passage obligé durant l'apprentissage du framework).

Même si l'ajout de style directement sur une balise est possible nativement en HTML, la syntaxe adopté par Tailwind CSS rend ce dernier beaucoup plus efficace en pratique :

On remarque ici que la syntaxe est bien plus efficiente à résultat équivalent.

Le framework se révèle donc d'une efficacité redoutable sur des projets complexes, et se prête à merveille à l'attribution de style situationnelle (on évite d'avoir à créer une classe ou un id pour un seul élément de notre projet), tout en conservant bien sur la possibilité de créer des classes personnalisées si un style est requis plusieurs fois.

Mise en place

Pour sa configuration, tailwind CSS nous offre plusieurs choix qui sont détaillés sur leur site:

L'ajout du script Play CDN

Il est possible d'utiliser le framework simplement en ajoutant un script spécifique au <head> de notre fichier HTML.

Cette facon de faire est cependant déconseillée, surtout si le projet à pour but de partir en production, la seconde méthode de configuration offre plus de flexibilité/controle et est plus adaptée le cas échéant. Cependant, l'ajout du script reste un moyen simple et rapide de découvrir le framework.

L'installation locale

Ici Tailwind CSS va venir installer les dépendances dont il a besoin pour fonctionner, directement dans notre projet. Cette installation peut se faire de plusieurs manières différentes, selon l'environnement de travail :

  • environnement Node.js
  • Stand-alone
  • En tant que plug-in PostCSS

De plus, une documentation est disponible pour faciliter l'installation et la configuration dans de multiples environnements de travail (laravel par exemple).

Comme évoqué plus haut, l'avantage d'une installation locale permet d'avoir plus de controle sur le framework, avec en particulier la possibilité, une fois le projet terminé, de purger les classes utilitaires qui ne sont pas utilisées, ou encore compacter notre code CSS pour réduire son nombre de lignes, afin d'envoyer en production le code le plus léger et optimisé possible.

Différences avec Bootstrap

Bootstrap est l'autre framework CSS le plus utilisé de nos jours, j'ai jugé pertinent de comparer les différences entre celui ci et Tailwind CSS, pour comprendre les forces et faiblesses de ce dernier face à un de ses concurrent.

Les deux frameworks fonctionnent principalement avec de l'intégration de code CSS directement dans les balises HTML via des classes pré-établies, mais la difference va se faire au niveau du contenu de ces classes.

Bootstrap va fonctionner autours de classes composants, qui vont être une associations de plusieurs directives CSS, qui auront pour résultat un design complet prêt à l'emploi.

Tailwind CSS se définit comme "utilities-first", ce qui se traduit par des classes utilitaires qui, comme expliqué plus haut, ne sont associés qu'a une seule directive CSS (une sorte de raccourci si l'on veut...).

Ainsi, là ou Bootstrap permet d'obtenir un résultat visuellement satisfaisant plus rapidement, Tailwind CSS reste nettement plus personnalisable dans sa conception.

Cependant, en 2020, les créateurs de Tailwind ont mis à disposition une offre payante nommée Tailwind UI qui propose un large choix de designs basées sur leur framework et prêts à l'emploi, ce qui peut combler les différences du framework face à d'autres comme Bootstrap.
